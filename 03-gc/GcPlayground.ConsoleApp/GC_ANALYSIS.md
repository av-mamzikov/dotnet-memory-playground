# Анализ поведения сборщика мусора .NET

## Выводы из RunGen0Test

### Основные принципы работы GC

1. **Поколенческая модель**
   - **Gen0**: новые объекты, собираются часто и быстро
   - **Gen1**: объекты, пережившие одну сборку Gen0
   - **Gen2**: долгоживущие объекты, собираются редко

2. **Счетчики сборок**
   - `GC.CollectionCount(generation)` показывает количество ВЫПОЛНЕННЫХ сборок
   - Не показывает количество объектов в поколении
   - Нулевые счетчики = сборки не происходили, а не объектов нет

### Поведение с разными размерами объектов

#### Маленькие объекты (< 85KB) - ИДЕАЛЬНЫЙ СЦЕНАРИЙ
```
✅ byte[1000] → Gen0 → быстрая уборка → минимальная нагрузка
```
- Объекты попадают в Gen0
- Gen0 сборки происходят часто, но очень быстро
- Производительность остается стабильной
- Низкое давление на память

#### Большие объекты (> 85KB) - ПРОБЛЕМНЫЙ СЦЕНАРИЙ
```
❌ byte[10MB] → LOH (Gen2) → редкая, дорогая уборка
```
- Объекты сразу попадают в LOH (Large Object Heap)
- LOH считается частью Gen2
- Требуют полной сборки мусора
- Высокое давление на память

### Практические рекомендации

#### ✅ Безопасные паттерны аллокации
- Мелкие временные объекты
- Объекты со сроком жизни < 1 секунды
- Временные переменные в методах
- Строковые операции, LINQ
- Временные коллекции

#### ❌ Проблемные паттерны
- Большие объекты в горячих путях кода
- Долгоживущие объекты в Gen1/Gen2
- Объекты, которые "живут" между сборками
- Бесконечный рост кэша без TTL

### Ключевые выводы

1. **Не бойтесь создавать временные объекты** - GC оптимизирован для этого
2. **Частые аллокации мелких объектов - это нормально**
3. **Следите за размером объектов** - порог LOH ~85KB
4. **Избегайте принудительного GC.Collect()** в продакшене
5. **Мониторьте Gen2 сборки** - они дорогие

### Метрики для мониторинга

- **Gen0 частота**: должна быть высокой (нормально)
- **Gen1 частота**: умеренная (ок)
- **Gen2 частота**: низкая (хорошо)
- **LOH размер**: минимальный (идеально)
- **Время пауз GC**: короткие (цель < 10ms для Gen0)

### Реальные примеры

#### Хороший паттерн:
```csharp
// Безопасно - мелкий, короткоживущий объект
var temp = new byte[1000];
ProcessData(temp);
// temp становится мусором, GC уберет в Gen0
```

#### Плохой паттерн:
```csharp
// Проблема - большой объект в цикле
for(int i = 0; i < 1000; i++) {
    var large = new byte[10 * 1024 * 1024]; // 10MB → LOH
    ProcessLargeData(large);
}
```

### Заключение

Система поколений GC в .NET очень эффективна для типичных паттернов аллокации. Создание множества мелких, короткоживущих объектов - это **идеальный сценарий**, для которого и был спроектирован сборщик мусора.
