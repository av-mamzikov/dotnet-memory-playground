## Эксперимент 3: Объекты размером 100KB (100000 байт) - КРИТИЧЕСКИЙ СЛУЧАЙ

### Данные:
```
Using size 100000
Iterations: 323, TotlSize: 32300000, Time: 5011ms, Gen0: 10 (each 3230000 byte), Gen1: 11, Gen2: 10, Memory: 0MB
Iterations: 644, TotlSize: 64400000, Time: 5007ms, Gen0: 20 (each 3220000 byte), Gen1: 21, Gen2: 20, Memory: 0MB
Iterations: 966, TotlSize: 96600000, Time: 5013ms, Gen0: 30 (each 3220000 byte), Gen1: 31, Gen2: 30, Memory: 1MB
Iterations: 1287, TotlSize: 128700000, Time: 5007ms, Gen0: 40 (each 3217500 byte), Gen1: 41, Gen2: 40, Memory: 1MB
Iterations: 1608, TotlSize: 160800000, Time: 5000ms, Gen0: 50 (each 3216000 byte), Gen1: 51, Gen2: 50, Memory: 1MB
Iterations: 1930, TotlSize: 193000000, Time: 5015ms, Gen0: 60 (each 3216666 byte), Gen1: 61, Gen2: 60, Memory: 1MB

```

### Выводы:
- **Gen0 ≈ Gen1 ≈ Gen2**: все счетчики почти одинаковые!
- **Очень частые сборки**: 10 Gen2 сборок за 5 секунд
- **Стабильный порог**: ~3.2MB между сборками
- **Низкое потребление памяти**: 0-1MB (агрессивная очистка)
- **ПРОБЛЕМА**: каждая сборка - это полная Gen2 сборка (медленная!)

### Почему это происходит:

#### Large Object Heap (LOH)
- Объекты >85,000 байт попадают в **LOH (Large Object Heap)**
- LOH является частью **Gen2**
- Объекты **сразу размещаются в Gen2**, минуя Gen0 и Gen1

#### Механика Gen2 сборок:
```
Обычный путь (<85KB):
Gen0 → (выжил) → Gen1 → (выжил) → Gen2

LOH путь (>85KB):
LOH (сразу Gen2) ❌
```

#### Почему Gen0 = Gen1 = Gen2:
- Каждая **Gen2 сборка автоматически включает Gen0 и Gen1**
- Это называется "full collection" (полная сборка)
- Поэтому все счетчики растут одновременно

## Производительность сборок

- **Gen0 сборка**: ~1мс (быстро, только молодые объекты)
- **Gen1 сборка**: ~5-10мс (умеренно, Gen0 + Gen1)
- **Gen2 сборка**: ~50-100мс+ (медленно, все поколения + LOH)

### Расчет для 100KB объектов:
- 10 Gen2 сборок за 5 секунд
- Каждая ~50-100мс
- **Потенциально 500-1000мс пауз за 5 секунд** = 10-20% времени в GC!

## Ключевые выводы

### 1. Порог Gen0 сборки
- Варьируется от **3-5MB** в зависимости от размера объектов
- Не фиксированное значение, зависит от эвристик GC

### 2. Критический порог 85KB
- Объекты >85KB попадают в LOH (Large Object Heap)
- LOH = Gen2 = дорогие полные сборки
- **Избегайте объектов >85KB в горячих путях кода**

### 3. Влияние размера объектов
- **Мелкие объекты (<10KB)**: идеальный сценарий для GC
- **Средние объекты (10-85KB)**: нормальное поведение
- **Большие объекты (>85KB)**: проблема производительности

### 4. Thread.Sleep влияет на результаты
- С `Thread.Sleep(1)` выделение памяти медленное
- В реальных приложениях без задержек Gen0 сборки будут чаще

## Практические рекомендации

### ✅ Делайте:
- Используйте мелкие временные объекты (<85KB)
- Переиспользуйте большие буферы через `ArrayPool<T>`
- Используйте `ObjectPool<T>` для дорогих объектов
- Мониторьте Gen2 сборки в продакшене

### ❌ Избегайте:
- Создания объектов >85KB в циклах
- Частого выделения больших массивов
- Долгоживущих объектов в Gen0/Gen1
- Принудительного `GC.Collect()` без необходимости

### Примеры оптимизации:

#### Плохо:
```csharp
for(int i = 0; i < 1000; i++) {
    var buffer = new byte[100_000]; // LOH!
    ProcessData(buffer);
}
```

#### Хорошо:
```csharp
var buffer = ArrayPool<byte>.Shared.Rent(100_000);
try {
    for(int i = 0; i < 1000; i++) {
        ProcessData(buffer);
    }
} finally {
    ArrayPool<byte>.Shared.Return(buffer);
}
```

## Заключение

Эксперименты наглядно показали:
1. GC оптимизирован для мелких короткоживущих объектов
2. Порог 85KB - критическая граница производительности
3. LOH требует особого внимания в высоконагруженных приложениях
4. Размер объектов напрямую влияет на частоту и стоимость GC сборок

**Главное правило**: держите объекты маленькими (<85KB) или переиспользуйте большие через пулы.
